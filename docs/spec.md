# spec

もう行けない・もう存在しない場所（倒産した店・取り壊された建物・廃線駅・失われた風景）の写真を皆で持ち寄り、十分に集まったら生成AIで蘇らせる、写真共有＋蘇生サービスの仕様。**懐かしむのではなく、蘇らせる**ことがミッション。

見た目（二面トグル・復元ゲージ・退色復活の見せ方）は `DESIGN.md`、コード構成・レイヤ分離・デプロイは `docs/architecture.md` を参照。ここに書くのは、コードが説明できない仕様（段階・エンゲージメント機構・データの扱い）。

## 段階（各段が単体で価値・各段の成果が次段の燃料）

蘇生は一足飛びではなく段階で積み上げる。各段はそれ自体で価値を持ち、その成果が次の段の燃料になる。

- **0段｜集める** — 写真を投稿する（退色復活・顔マスク・捕獲を伴う）。燃料はデータ。単体で「郷愁アーカイブ」として価値があり、これだけでも出せる。
- **1段｜幻想モード** — 同じ場所を「冷たい証拠」と「AI再現」でトグルする。証拠⇄再現の行き来。
- **2段｜再現** — 多角の写真から三面図・フィギュア絵を生成する。
- **3段｜物理化** — 3Dモデルから鉄道模型を 3Dプリントする。
- **4段｜VR** — 蘇った場所を歩ける。

再現は集めたデータ無しには成立しない。だから **0段は劣化版ではなく、夢の土台そのもの**。壮大な夢ほど、最初の一段を早く出す理由になる。

## エンゲージメント機構

### 復元ゲージ

- 場所ごとに、必要な**面／階／角度**の充足度を可視化する（例: 北面✓ 南面✓ 東面✗ 内部2F✗ …幻想モードまであと○面）。
- 満ちると**幻想モードを解放**する（＝蘇生生成）。これが普通の写真SNSにない**勝利条件**。
- ユニットは「写真の枚数」でなく「**場所の蘇生進捗**」。

### 「写りが悪くてもいい」

- 欠けた面のピンボケ一枚も、AI再現の形状・色を縛る**材料**になる。写真は鑑賞物でなくデータ。
- **完成度 ＞ 美しさ**の価値逆転により、投稿の敷居を下げる。

### 貢献度は「量」でなく「価値」で測る

- 欠けた面を埋めた一枚 ≫ 既出面の二枚目。再撮影不能な一枚 ≫ 今すぐ撮れる楽な一枚。
- **しきい越え**（最後の一片で幻想モードを解放した人）に特別な栄光を与える。
- こうして「一気に貴重な写真を供給した人」の称号が本物になる（量の王ではなく、二度と手に入らない材料の供給者）。

### 称号体系

- アプリ＝**Re:Render** ／ 住民＝**Re:Renderer** ／ 頂点＝**幻想神**（貢献種別ごとの階位）。

## 幻想モード（1段の中身）

幻想モードは「再現を眺める」画面ではなく、**入ると世界が一変する没入モード**。

- 入ると、灰色の証拠（トリック面）から、**色と人のいる往時**（幻想面）へ一変する。
- 蘇った場所を低摩擦で歩き・触り・手を入れられる（思考＝操作の没入）。
- 蘇った建物の什器・看板・什物がキャラ化した**眷属**を、その場所固有の存在として従えられる。往時の空気を語り部として宿す。
- 幻想を与える存在（ホスト／ナビ役）を置ける。

「証拠 ⇄ 往時」を同じ場所で行き来できることが、ただのアーカイブと一線を画す核。

## 退色復活（記録の回復であって捏造ではない）

- クライアント側の古典的画像処理（色かぶり除去・ホワイトバランス／コントラスト回復）で、褪せた写真を薄く回復する。
- AIによる着色は後段の重い処理として後付け可能。ただし **「復元（元情報の回復）」と「再構成（無い色をAIが推測して足すこと）」は別物**として扱い、再構成部分は明示する。ミッション（記録を蘇らせる）と自己矛盾しないための**製品規律**。
- **記録と再構成を混同させない**——このアプリが「捏造した過去」を配らないための一線。

## 顔マスク（privacy・肖像権）

- 古い公共写真には同意なき他人の顔が写る。投稿時にクライアント側で顔検出→ぼかしを**既定寄り**に置く。
- 単なるフィルタでなく、公開・永続保管を続けるための**責任の裏づけ**。

## 捕獲（私的コールド保管）

- 永続性が製品本体（対象は再撮影不能）なので、投稿時の**画像実体を継続保管**する（後からは取れない時間非対称性）。
- クライアント側で完結させる（サーバ側の常駐ML処理を避ける＝土台の薄さを保つ）。デッドリンク置換ロジックは先回りせず、必要になってから足す。

## 権利のリスクは段が上がるほど上がる

- 消えた場所・非営利の閲覧は低リスク。段（物理化・物販等）を上がるほど、建築著作権や商標のリスクが上がる。仕様上、各段のリスク段階を意識して機能を分ける。

## 検証したいこと（テスト観点）

- 復元ゲージの判定: 任意の投稿集合（どの面/階/角度が埋まったか）に対して、幻想モード解放の閾値判定が意図通り動くか。
- 定義データ（場所定義・写真メタ）と実行時状態（復元進捗・幻想モード状態）が混ざっていないか。
- 退色復活（記録回復）とAI着色（再構成）が、データ上・表示上で区別されているか。
- 顔マスクが既定で効き、投稿経路で外れないか。

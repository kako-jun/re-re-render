# architecture

見た目は `DESIGN.md`、段階・エンゲージメント機構・データの扱いは `docs/spec.md` を参照。ここはコード構成・レイヤ分離・技術スタック・デプロイ。

## 出自 — hanoba のフォーク

- **hanoba（Nostr 基盤の写真共有クライアント）をフォークし、被写体を差し替える**ことから始める。
- hanoba からの差分 ＝ **−レトロガチャ ＋退色復活 ＋顔マスク ＋捕獲**。
  - **−レトロガチャ**: hanoba のランダム加工ガチャは持ち込まない。
  - **＋退色復活**: 褪せた写真の色かぶり除去・WB/コントラスト回復。
  - **＋顔マスク**: 同意なき他人の顔の検出→ぼかし（既定寄り）。
  - **＋捕獲**: 投稿時の画像実体の継続保管（私的コールド保管）。
- ただし製品人格は裏返る（現在を飾るのではなく、過去を蘇らせ・守り・残す）。上層（AI再現・3D・VR）で構想は大きく広がる。

## 土台は薄く・上層は重く（意識的に分ける）

- **土台（0段：集める）は薄く保つ**。退色復活・顔マスク・捕獲はいずれも**クライアント側で完結**させ、サーバ側の常駐ML処理を持たない（薄さの維持）。
- **上層（1〜4段：AI再現／三面図／3Dモデル／VR）は重く自前**。ここはバックエンドレスでは成立しないため、土台とは別の層として設計し、混ぜない。

## スタック

- ベースは hanoba を継承（Nostr 基盤・静的＋island 系の Web クライアント・PWA・CF Pages デプロイを想定）。正確なフレームワーク・リレー構成は着手時に hanoba の現行実装を読んで確定する。
- 本文・投稿は Nostr のイベントとして扱い、独自の本文ストレージ・鍵管理を抱え込まない方針（hanoba の姿勢を継承）。

## レイヤ分離

蘇生パイプラインを、責務ごとに三層に分ける。相互依存を網状に広げない。

```
src/
  nostr/            # hanoba 流用層: Nostr 購読・投稿・署名検証。リレー接続の唯一の窓口
  revive/           # 蘇生エンジン層: 退色復活・顔マスク・捕獲（クライアント側完結）
    fade-recovery   #   退色復活（色かぶり除去・WB/コントラスト回復）＝記録の回復
    face-mask       #   顔検出→ぼかし（既定寄り）
    capture         #   画像実体の私的コールド保管
  genso/            # 幻想モード表示層: 証拠⇄往時のトグル、復元ゲージ、再構成の明示
  state/            # アプリ側の派生状態（復元進捗・幻想モード状態）。クライアント側に閉じる
```

- **hanoba 流用層 / 蘇生エンジン層 / 幻想モード表示層を分離する**。蘇生エンジン層は画像処理の純粋関数（入力画像→加工結果）に切り出し、DOM/UI に依存させない（テスト可能性）。
- **「復元（元情報の回復）」と「再構成（AIの推測補完）」をデータ上で区別する**。退色復活・古典処理は復元、AI着色・欠損面の推測生成は再構成。両者を同じフラグ・同じ型で混同させない（製品規律）。
- 復元ゲージの充足判定（どの面/階/角度が埋まったか→幻想モード解放閾値）は、UI から独立した純粋関数にしてテスト可能にする。

## 定義データと実行時状態の分離

- **定義（不変）**: 場所定義（対象の場所・必要な面/階/角度の一覧）、投稿された写真のメタ（撮影角度・面・退色/再構成の別）。
- **実行時状態（可変）**: 復元進捗（充足した面の集合）、幻想モードの解放フラグ・表示状態。
- 定義側に進捗を焼き込まない・状態側に場所仕様を焼き込まない。派生状態はサーバに持たせず、クライアント側で完結させる（土台の薄さを保つ）。

## 上層（AI再現／3D／VR）の切り分け

- AI着色・多角写真からの三面図/3Dモデル生成・VR化は、土台とは別の重い処理層。**後付けの後段パイプライン**として分離し、0段の薄い土台を汚さない。
- これらは非同期の重処理（GPU等）を要するため、クライアント側完結の原則の外に置く。実装方式は段階着手時に確定する。
